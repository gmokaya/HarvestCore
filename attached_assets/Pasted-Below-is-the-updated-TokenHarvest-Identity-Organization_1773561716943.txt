# TokenHarvest Identity & Organization Hierarchy Model (Including Warehouses)

---

# 1. Core Platform Hierarchy

The platform hierarchy includes **organizations and physical warehouse infrastructure**.

```text
TokenHarvest Platform
│
├ Parent Organization
│   │
│   ├ Subsidiary Organization
│   │   ├ Users
│   │   ├ Warehouses
│   │   └ Assets
│   │
│   ├ Cooperative / Farmer Group
│   │   ├ Farmers
│   │   └ Commodity Deposits
│   │
│   └ Processing Company
│       ├ Buyers
│       └ Procurement Teams
│
└ Warehouse Network
    ├ Warehouse Operator Organization
    │   ├ Warehouse 1
    │   ├ Warehouse 2
    │   └ Warehouse 3
```

This structure ensures **warehouses are clearly linked to organizations but remain independent operational entities**.

---

# 2. Organization Types

Organizations can represent **different participants in the commodity ecosystem**.

| Organization Type        | Description                     |
| ------------------------ | ------------------------------- |
| Individual               | Single farmer or trader         |
| Cooperative              | Farmer cooperative              |
| Warehouse Operator       | Warehouse management company    |
| Processor / Manufacturer | Commodity processing companies  |
| Trader / Aggregator      | Commodity traders               |
| Financial Institution    | Banks or lenders                |
| Investor                 | Liquidity providers             |
| Platform Admin           | TokenHarvest platform operators |

Example:

```
Organization: TransNzoia Storage Ltd
Type: Warehouse Operator
```

---

# 3. Warehouse Entity

Warehouses are **separate operational units linked to organizations**.

Example:

```
Organization: TransNzoia Storage Ltd
│
├ Warehouse Kitale
├ Warehouse Endebess
└ Warehouse Eldoret
```

Each warehouse can:

* receive commodities
* issue warehouse receipts
* conduct inspections
* manage storage capacity

---

# 4. Warehouse Data Structure

Each warehouse must store the following information.

| Field           | Description                   |
| --------------- | ----------------------------- |
| warehouse_id    | Unique warehouse identifier   |
| organization_id | Owner organization            |
| warehouse_name  | Name of warehouse             |
| location        | Physical location             |
| capacity        | Storage capacity              |
| warehouse_type  | Grain / Tea / Multi-commodity |
| status          | Active / Suspended            |

Example:

```json
{
 "warehouse_id": "WH-KTL-001",
 "organization_id": "ORG-200",
 "warehouse_name": "Kitale Central Warehouse",
 "location": "Kitale, Kenya",
 "capacity": "20000 tons",
 "warehouse_type": "Grain",
 "status": "Active"
}
```

---

# 5. Warehouse Roles

Users can belong to a warehouse.

Example roles:

| Role                     | Function                      |
| ------------------------ | ----------------------------- |
| Warehouse Manager        | Oversees warehouse operations |
| Inspection Officer       | Performs quality inspection   |
| Commodity Intake Officer | Records commodity deposits    |
| Warehouse Auditor        | Verifies stock levels         |

Example hierarchy:

```
Warehouse Kitale
│
├ Warehouse Manager
├ Inspection Officers
└ Commodity Intake Officers
```

---

# 6. Warehouse Access Control

Warehouse users can **only access data for their warehouse**.

Example rule:

```
User.warehouse_id = Data.warehouse_id
```

If mismatch:

```
Access denied
```

---

# 7. Warehouse and Commodity Relationship

Each warehouse manages **commodity batches**.

Example:

```
Warehouse Kitale
│
├ Commodity Batch 1 – Maize
├ Commodity Batch 2 – Beans
└ Commodity Batch 3 – Green Gram
```

Commodity batches link to:

* inspection reports
* warehouse receipts
* tokenized assets

# 8. Warehouse and Receipt System

When commodities are deposited:

```
Commodity Intake
      ↓
Inspection
      ↓
Digital Warehouse Receipt
      ↓
Tokenization
```

Example receipt:

```
Receipt ID: WR-2026-0042
Warehouse: WH-KTL-001
Commodity: White Maize
Quantity: 50 Tons
Owner: Farmer_0092
```

# 11. Data Segregation Model

Every record contains:

```
organization_id
warehouse_id
```

Access rules:

```
User.organization_id = Data.organization_id
AND
User.warehouse_id = Data.warehouse_id
```

Parent organizations may access **all child warehouses**.
